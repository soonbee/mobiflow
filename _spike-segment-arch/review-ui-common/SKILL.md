# Common UI Review

플랫폼(웹/모바일)에 관계없이 UI 코드에서 공통으로 점검해야 할 항목을 다룬다.
백엔드 연결이나 비즈니스 로직은 범위 밖이다. 순수하게 화면에 보이는 것, 사용자가 조작하는 것에 집중한다.

파일을 수정하지 않는다. 발견 사항만 보고한다.

## Review Process

1. 변경된 UI 관련 파일 목록을 확인한다 (`git diff HEAD~1 --name-only`로 시작, 범위는 지시에 따라 조정)
2. 변경 내용을 확인한다 (`git diff HEAD~1`)
3. 아래 체크리스트 각 항목에 대해 점검한다
4. 우선순위별로 발견 사항을 보고한다

## Review Checklist

### 1. 컴포넌트 설계 — 재사용성과 단일 책임

컴포넌트가 잘 분리되어 있을수록 수정 범위가 좁아지고 테스트가 쉬워진다. 하나의 컴포넌트가 여러 역할을 맡으면 변경 시 의도치 않은 곳이 깨진다.

- **단일 책임**: 하나의 컴포넌트가 하나의 역할만 담당하는가? 200줄이 넘는 컴포넌트는 분리 후보다.
- **재사용 가능한 공통 요소**: 버튼, 입력 필드, 카드 등 반복 사용되는 UI가 공통 컴포넌트로 추출되어 있는가? 같은 스타일의 버튼을 3곳 이상에서 직접 만들고 있다면 지적하라.
- **Props/Parameter 설계**: props가 너무 많으면(7개 이상) 컴포넌트가 너무 많은 일을 하고 있다는 신호다. 반대로 하드코딩된 값이 props로 빠져야 할 수도 있다.
- **컴포넌트 깊이**: 불필요한 wrapper 컴포넌트가 없는가? 반대로 한 컴포넌트 안에 모든 것이 flat하게 들어있어서 구조 파악이 어렵지는 않은가?

```
# Bad — 하나의 컴포넌트가 헤더, 본문, 푸터를 모두 포함
<ProductPage>
  <!-- 500줄의 JSX -->
</ProductPage>

# Good — 역할별로 분리
<ProductPage>
  <ProductHeader />
  <ProductDetails />
  <ProductReviews />
  <ProductFooter />
</ProductPage>
```

### 2. 상태 관리 — 모든 화면 상태가 커버되는가

사용자가 실제로 마주칠 수 있는 모든 상태가 UI에 반영되어야 한다. 누락된 상태는 빈 화면이나 깨진 레이아웃으로 이어진다.

- **필수 상태 커버리지**: 아래 상태들이 모두 처리되어 있는가?
  - 로딩 중 (skeleton, spinner 등)
  - 데이터 없음 (empty state)
  - 에러 발생
  - 정상 표시
  - (해당 시) 부분 로딩, 페이지네이션, 무한스크롤 끝

- **상태 위치**: 상태가 필요 이상으로 상위 컴포넌트에 올라가 있지 않은가? 특정 컴포넌트 내부에서만 쓰이는 상태는 해당 컴포넌트에 두는 것이 맞다.
- **파생 상태**: 다른 상태에서 계산 가능한 값을 별도 상태로 관리하고 있지 않은가? (예: `items`와 `itemCount`를 둘 다 상태로 관리)

```
# Bad — 로딩과 에러 상태 누락
function UserList({ users }) {
  return users.map(u => <UserCard user={u} />);
}

# Good — 모든 상태 커버
function UserList({ users, isLoading, error }) {
  if (isLoading) return <Skeleton />;
  if (error) return <ErrorMessage error={error} />;
  if (users.length === 0) return <EmptyState />;
  return users.map(u => <UserCard key={u.id} user={u} />);
}
```

### 3. 스타일링 일관성 — 디자인 시스템과의 정합성

매직 넘버와 하드코딩된 색상은 디자인 변경 시 누락의 원인이 된다. 토큰이나 변수로 관리하면 한 곳만 바꿔도 전체에 반영된다.

- **디자인 토큰 사용**: 색상, 간격, 폰트 크기, 그림자 등이 디자인 시스템 토큰(또는 테마 변수)을 사용하고 있는가?
- **매직 넘버**: `padding: 13px`, `color: #3a7bd5` 같은 하드코딩 값이 있다면 의도된 것인지 확인하라.
- **일관된 간격 체계**: 4px 또는 8px 기반 간격 체계를 따르고 있는가? 불규칙한 간격(`margin: 7px`, `padding: 11px`)은 지적하라.
- **z-index 관리**: z-index가 임의의 큰 숫자(`z-index: 9999`)로 설정되어 있다면 체계적인 관리가 필요하다.

### 4. 접근성 (a11y) — 모든 사용자를 위한 UI

접근성은 법적 요구사항이기도 하지만, 키보드 사용자, 스크린 리더 사용자, 시각 장애인 등 더 넓은 사용자층을 지원하기 위함이다.

- **시맨틱 마크업**: `<div>`와 `<span>`으로만 구성되어 있지 않은가? `<button>`, `<nav>`, `<main>`, `<section>`, `<header>` 등 적절한 시맨틱 태그를 사용하라.
- **인터랙티브 요소**: 클릭 가능한 모든 요소가 키보드로도 접근 가능한가? `<div onClick>` 대신 `<button>`을 사용하고 있는가?
- **ARIA 속성**: 커스텀 위젯(드롭다운, 모달, 탭 등)에 적절한 aria 속성(`aria-label`, `aria-expanded`, `role` 등)이 있는가?
- **색상 대비**: 텍스트와 배경의 대비가 WCAG AA 기준(4.5:1)을 충족하는가?
- **이미지 대체 텍스트**: 모든 의미 있는 이미지에 alt 텍스트가 있는가? 장식용 이미지는 `alt=""`로 처리되었는가?
- **포커스 관리**: 모달이 열릴 때 포커스가 모달로 이동하는가? 닫힐 때 원래 위치로 돌아오는가?

```
# Bad — 접근성 없는 클릭 요소
<div className="card" onClick={handleClick}>
  <img src={product.image} />
  <span>{product.name}</span>
</div>

# Good — 시맨틱하고 접근 가능한 구조
<article className="card">
  <button onClick={handleClick} aria-label={`${product.name} 상세 보기`}>
    <img src={product.image} alt={product.name} />
    <h3>{product.name}</h3>
  </button>
</article>
```

### 5. 렌더링 성능 — 불필요한 작업 줄이기

불필요한 렌더링은 화면 버벅임과 배터리 소모로 이어진다. 특히 리스트와 애니메이션에서 중요하다.

- **불필요한 리렌더링**: 부모 상태 변경 시 관련 없는 자식까지 리렌더링되고 있지 않은가? 메모이제이션(React.memo, useMemo, useCallback 등)이 필요한 곳에 적용되어 있는가?
- **리스트 키**: 리스트 렌더링 시 고유한 key가 사용되는가? `index`를 key로 사용하면 항목 순서 변경 시 문제가 발생한다.
- **큰 리스트 가상화**: 100개 이상의 항목을 렌더링하는 리스트가 가상화(virtualization) 없이 전체 렌더링되고 있지 않은가?
- **이미지 최적화**: 적절한 크기와 포맷의 이미지를 사용하고 있는가? lazy loading이 적용되어 있는가?
- **애니메이션**: CSS transform/opacity를 사용하는가, 아니면 layout을 유발하는 속성(width, height, top, left)을 애니메이션하고 있는가?

### 6. 네이밍과 파일 구조 — 찾기 쉽고 예측 가능한 코드

이름만으로 역할을 파악할 수 있어야 한다. 6개월 후 다른 개발자가 코드를 봤을 때 폴더와 파일명만으로 구조를 이해할 수 있는가?

- **컴포넌트 이름**: 역할을 명확히 드러내는가? `Component1`, `Container`, `Wrapper` 같은 모호한 이름은 지적하라.
- **Props/변수 이름**: boolean props는 `is-`, `has-`, `should-` 접두사를 사용하는가? 이벤트 핸들러는 `on-`, `handle-` 접두사를 사용하는가?
- **파일/폴더 구조**: 팀 컨벤션에 맞는가? 관련 파일(컴포넌트, 스타일, 테스트)이 가까이 위치하는가?

## Output Format

발견 사항을 아래 우선순위로 정리하여 보고한다.

**🔴 Critical (반드시 수정):**
다른 컴포넌트로 퍼지면 수정 비용이 급증하는 문제. 공통 컴포넌트의 잘못된 인터페이스, 접근성 미준수, 필수 상태 누락.

**🟡 Warning (수정 권장):**
코드 품질을 저해하지만 즉시 퍼지지는 않는 문제. 매직 넘버, 과도한 리렌더링, 모호한 네이밍.

**🟢 Suggestion (고려):**
개선하면 좋지만 필수는 아닌 항목. 컴포넌트 분리 기회, 스타일 정리, 사소한 구조 개선.

각 항목에 대해:

- 파일명과 위치
- 현재 코드 인용
- 구체적인 개선안 제시
- 왜 지금 고쳐야 하는지(퍼짐 위험) 또는 나중에 고쳐도 되는지 판단 근거

발견 사항이 없으면 간결하게 "이상 없음"으로 보고한다.
