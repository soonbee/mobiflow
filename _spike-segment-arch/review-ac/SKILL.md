# review-ac

티켓의 **Acceptance Criteria**와 **완료 기준**에 선언된 요구사항이 실제 구현에 모두 반영되었는지 점검한다.
코드 품질·UI 품질 점검과는 관점이 다르다. 여기서는 "무엇을 구현해야 했나"와 "무엇이 구현되었나"를 비교한다.

파일을 수정하지 않는다. 발견 사항만 보고한다.

---

## 입력

리뷰어 에이전트는 반드시 아래를 컨텍스트로 받아야 한다. 누락 시 AC 점검 불가.

- **티켓 파일 경로** (예: `docs/tickets/v0.2.0/007.md`)
- 구현 대상 브랜치 또는 worktree (리뷰어가 동작 중인 위치)

티켓 경로가 전달되지 않으면 다음과 같이 보고하고 AC 점검을 스킵한다:

```
⚠️ AC 점검 스킵: 티켓 파일 경로가 전달되지 않음. 호출자(러너)에서 티켓 경로를 명시 전달 필요.
```

---

## Review Process

1. 티켓 파일을 읽어 다음 섹션을 파싱한다:
   - `## Acceptance Criteria` — 체크박스 `- [ ]` 목록 (주 파싱 대상)
   - `## 완료 기준` — 체크박스 `- [ ]` 목록 (보조, 참고)
   - `## 파일 스코프 (Scope)` — 수정 대상·금지·신규 생성 경계
   - `## 참조 (Reference)` — 시안·명세 링크 (엣지케이스 추론 힌트)
2. 구현 변경분을 확인한다:
   ```bash
   git diff develop...HEAD --stat
   git diff develop...HEAD --name-only
   ```
3. 각 AC 항목에 대해 구현 위치를 추적한다. `grep -rn`으로 AC가 언급하는 키워드(API 경로, 컴포넌트명, 상태명, 메시지 등)를 코드에서 검색한다.
4. 아래 체크리스트로 누락·엣지케이스·범위 일탈을 점검한다.
5. AC 매핑 표와 우선순위별 발견 사항을 보고한다.

---

## Review Checklist

### 1. AC 매핑 — 항목별 구현 존재 여부

- `## Acceptance Criteria`의 각 체크박스 항목이 코드·파일·테스트 중 어디에 반영되었는지 위치를 특정한다
- 위치를 특정할 수 없는 항목은 미구현으로 간주한다
- 부분 구현(조건 일부만 충족)은 경고로 분리한다

### 2. 누락 검출 — 티켓에 있지만 코드에 없는 것

- 티켓에 명시된 동작·상태·경로 중 구현 변경분에서 발견되지 않는 항목
- 특히 체크하기 쉽지만 놓치기 쉬운 것:
  - 에러 메시지 문구
  - 빈 상태(empty state) 텍스트·이미지
  - 로딩 상태 인디케이터
  - 성공/실패 피드백 (토스트, 알림 등)
  - 권한·로그인 상태에 따른 분기 노출

### 3. 엣지케이스 — AC가 암시하거나 명시한 예외 경로

- 에러 상태: 네트워크 실패, 서버 4xx/5xx, 타임아웃
- 빈 상태: 데이터 없음, 검색 결과 없음
- 경계 케이스: 첫 번째·마지막 위치, 단일 항목, 최대 길이
- 상태 전이 시점: "변경 직후(서버 응답 전)"과 "재진입(새로고침) 후" (티켓 AC 작성 기준에 명시)

### 4. 조건부 노출 — 특정 조건에서만 동작해야 하는 것

- 로그인 여부, 권한, 플랜, 기능 플래그
- 특정 상태(예: 편집 중에만, 관리자에게만)
- 플랫폼/디바이스 분기(해당 시)

### 5. 파일 스코프 준수

- 구현 변경분이 `## 파일 스코프`의 **수정 대상** 안에 있는가
- **수정 금지** 경로를 건드리지 않았는가
- **신규 생성**으로 선언되지 않은 파일이 추가되지 않았는가
- 스코프 일탈은 수정 필요가 아니라 주의 플래그로 보고

### 6. 완료 기준 (보조)

- `## 완료 기준`의 체크박스 중 AC 외의 항목(테스트, 문서화 등)은 참고로만 확인
- 러너의 다른 단계(static-fixer 등)가 이미 담당하는 항목은 중복 지적하지 않는다

## Output Format

리뷰어 에이전트의 출력에서 **Part B — AC 점검** 섹션으로 다음 형식을 사용한다.

```
## Part B — AC 점검 (review-ac 기준)

티켓: docs/tickets/v{X.Y.Z}/{N}.md — {제목}

### AC 매핑

| # | AC 항목 | 구현 위치 | 상태 |
|---|---------|-----------|------|
| 1 | {AC 요약} | {path:line 또는 파일명} | ✅ |
| 2 | {AC 요약} | — | ❌ |
| 3 | {AC 요약} | {path:line} | ⚠️ 부분 |

### 🔴 수정 필요 (Must Fix)

- **AC-2 미구현**: {설명. 어디에 추가되어야 하는지 단서가 있으면 포함}
- **누락**: {티켓이 명시했으나 구현에 없는 항목}

### 🟡 권장 (Should Fix)

- **엣지케이스 누락 의심**: {설명}
- **조건부 노출 불명확**: {설명}

### 🟢 통과 (Pass)

- {AC-N 충족 간단 요약}

### ⚠️ 주의 플래그 (참고)

- **파일 스코프 일탈**: {경로} (수정 금지 영역 또는 미선언 신규 파일)

### 요약

- AC 충족: {M}/{N}
- 수정 필요: {K}건 | 권장: {L}건 | 주의: {P}건
```

---

## 주의사항

- **관점 분리**: 품질(Part A)과 AC(Part B)를 섞지 않는다. 같은 이슈가 두 관점 모두에서 걸리면 각각의 파트에 다른 각도로 기술
- **구현 추적 실패 ≠ 미구현**: 키워드로 찾지 못했다면 티켓의 AC가 추상적일 수 있다. 구현 위치 불명 상태로 보고하고 리뷰어 판단을 청한다
- **티켓 수정 금지**: 이 스킬은 티켓 파일을 읽기만 한다. AC가 불명확해도 티켓을 고치지 않는다
- **러너 외 단계 중복 점검 금지**: static-fixer / reviewer-ui-\* 가 이미 담당하는 항목은 다루지 않는다. 여기서는 AC 충족에만 집중
