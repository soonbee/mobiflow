# domain-contract

코드를 구현할 때 drift(요구사항 이탈)를 방지하고 품질을 확보하기 위한 워크플로우. AI 에이전트가 긴 코드를 생성하면서 초기 요구사항에서 벗어나는 경향을 계약과 테스트로 앵커링한다.

## 테스트 계층

이 워크플로우의 테스트는 **계약 테스트**다. "무엇을 해야 하는가"를 표현하며, 요구사항이 바뀌면 함께 업데이트한다.

구현 완료 후 커버리지를 보완하는 **백필 테스트**는 본 스킬의 범위 밖. integration-verify phase의 `with-tests-backfill`이 담당.

계약 테스트와 백필 테스트는 파일 또는 네이밍으로 구분한다. 예: `order.contract.test.ts` vs `order.backfill.test.ts`, 또는 프로젝트 컨벤션에 맞는 방식.

## 신규 구현 워크플로우

```
요구사항 분석 → 계약 정의 → 계약 테스트 → 구현 → 테스트 실행·수정 루프
```

### Phase 1: 요구사항 분석

구현 요청을 받으면 코드를 바로 작성하지 않는다. 먼저 다음을 파악한다:

1. **입력과 출력**
2. **부수 효과**
3. **실패 조건**
4. **호출자 맥락**

프로젝트에 기존 코드가 있으면 관련 파일을 읽어서 언어, 테스트 프레임워크, 타입 시스템, 네이밍 컨벤션을 확인한다.

### Phase 2: 계약(Contract) 정의

로직은 비워두고 **형태만** 선언한다.

계약에 포함할 것:

- 함수/메서드 시그니처 (이름, 파라미터, 반환 타입)
- 타입/인터페이스 정의
- 핵심 불변조건을 주석으로 명시
- 에러 타입 또는 예외 클래스
- public API 경계

### Phase 3: 계약 테스트

모든 케이스를 테스트하지 않는다. 구현의 방향을 잡아주는 나침반 역할만 한다.

- Happy path 1개
- 위험한 edge case 2~3개 (데이터 무결성, 불변조건 위반, 상태 변경 우선)

총 3~4개. 5개를 넘기지 않는다.

### Phase 4: 구현

계약에서 정의한 시그니처와 타입을 그대로 사용하여 실제 로직을 채운다.

- 계약의 시그니처를 변경하지 않는다. 변경이 불가피하면 계약과 계약 테스트를 먼저 업데이트한다.
- 불변조건 주석에 명시된 동작을 반드시 구현한다.

### Phase 5: 테스트 실행·수정 루프

계약 테스트를 실행한다. 실패가 있으면 수정한다.

**핵심 규칙: 테스트가 아니라 코드를 수정한다.**

테스트를 수정해도 되는 유일한 경우:

- 테스트 자체에 버그가 있는 경우 (잘못된 기댓값, 오타)
- 요구사항이 변경되어 테스트가 더 이상 유효하지 않은 경우

수정 후 다시 전체 테스트를 실행한다. 모든 테스트가 통과할 때까지 반복한다.

## 기존 기능 수정 워크플로우

```
요구사항 분석 → 계약 업데이트 → 계약 테스트 업데이트 → 구현 수정 → 테스트 실행·수정 루프
```

신규 구현과의 차이:

1. **계약과 계약 테스트를 먼저 업데이트한다.** 변경된 요구사항을 계약과 계약 테스트에 반영한 뒤 구현을 수정한다.
2. **백필 테스트는 건드리지 않는다.** integration-verify phase의 `with-tests-backfill` 책임.

## 탐색적 작업에서의 적용

프로토타이핑, 새 API 실험 등 "올바른 결과"가 아직 불분명한 상황에서는 전체 워크플로우를 강제하지 않는다.

최소 적용:

- Phase 2에서 함수 시그니처와 반환 타입만 계약으로 선언한다
- Phase 3은 건너뛴다
- Phase 4에서 구현한다
- 프로토타입이 안정되면 Phase 3, 5를 소급 적용한다

## smoke와의 관계

테스트 실행은 `npm run smoke`(또는 `smoke:min`)에 포함되므로 본 스킬에서 작성한 계약 테스트는 자연스럽게 smoke 게이트를 통과해야 한다. test runner 미설치 프로젝트라면 별도 인프라 티켓.
