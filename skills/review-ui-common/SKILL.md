# review-ui-common

플랫폼(웹/모바일)에 관계없이 UI 코드에서 공통으로 점검해야 할 항목. `code-reviewer` Part A의 핵심 체크리스트 중 하나.

백엔드 연결이나 비즈니스 로직은 범위 밖. 순수하게 화면에 보이는 것, 사용자가 조작하는 것에 집중한다. **파일을 수정하지 않는다.**

## Review Process

1. 변경된 UI 관련 파일을 식별 (`git diff develop..HEAD --name-only`)
2. 변경 내용 확인 (`git diff develop..HEAD`)
3. 아래 체크리스트 각 항목 점검
4. 우선순위별로 발견 사항을 보고 (🔴/🟡/pass)

## Review Checklist

### 1. 컴포넌트 설계 — 재사용성과 단일 책임

- **단일 책임**: 하나의 컴포넌트가 하나의 역할만 담당하는가? 200줄 넘는 컴포넌트는 분리 후보
- **재사용 가능한 공통 요소**: 버튼, 입력 필드, 카드 등 반복 사용되는 UI가 공통 컴포넌트로 추출되어 있는가? 같은 스타일을 3곳 이상에서 직접 만들고 있다면 지적
- **Props 설계**: props가 7개 이상이면 컴포넌트가 너무 많은 일을 하고 있다는 신호. 반대로 하드코딩 값이 props로 빠져야 할 수도 있음
- **컴포넌트 깊이**: 불필요한 wrapper가 없는가? 반대로 한 컴포넌트에 모든 것이 flat하게 들어있어 구조 파악이 어렵지 않은가?

### 2. 상태 관리 — 모든 화면 상태가 커버되는가

사용자가 마주칠 수 있는 모든 상태가 UI에 반영되어야 한다.

- **필수 상태 커버리지**: 로딩 / 데이터 있음 / 데이터 없음 / 에러 / (해당 시) 부분 로딩, 페이지네이션 끝
- **상태 위치**: 상태가 필요 이상으로 상위에 올라가 있지 않은가
- **파생 상태**: 다른 상태에서 계산 가능한 값을 별도 상태로 관리하지 않는가 (예: `items`와 `itemCount` 둘 다)

### 3. 스타일링 일관성 — 디자인 시스템과의 정합성

- **디자인 토큰 사용**: 색상·간격·폰트 크기·그림자가 토큰(또는 테마 변수)을 사용하는가
- **매직 넘버**: `padding: 13px`, `color: #3a7bd5` 같은 하드코딩 값이 의도된 것인지
- **일관된 간격 체계**: 4px/8px 기반 간격 체계 준수
- **z-index 관리**: 임의의 큰 숫자(`z-index: 9999`) 대신 체계적 관리

### 4. 접근성 (a11y)

- **시맨틱 마크업**: `<button>`, `<nav>`, `<main>`, `<section>`, `<header>` 등 적절히 사용
- **인터랙티브 요소**: 클릭 가능한 모든 요소가 키보드 접근 가능 (`<div onClick>` 대신 `<button>`)
- **ARIA 속성**: 커스텀 위젯에 `aria-label`, `aria-expanded`, `role` 등
- **색상 대비**: 텍스트와 배경 대비 WCAG AA(4.5:1) 충족
- **이미지 대체 텍스트**: 의미 있는 이미지에 alt, 장식용은 `alt=""`
- **포커스 관리**: 모달 열기/닫기 시 포커스 이동 적절

### 5. 렌더링 성능

- **불필요한 리렌더링**: 부모 상태 변경 시 무관한 자식까지 리렌더되지 않는지. 메모이제이션 필요한 곳에 적용
- **리스트 키**: 고유 key 사용. `index`를 key로 쓰면 항목 순서 변경 시 문제
- **큰 리스트 가상화**: 100개 이상 항목을 virtualization 없이 전체 렌더링하지 않는지
- **이미지 최적화**: 적절한 크기·포맷 + lazy loading
- **애니메이션**: CSS transform/opacity vs layout 유발 속성 (width/height/top/left)

### 6. 네이밍과 파일 구조

- **컴포넌트 이름**: 역할 명확. `Component1`, `Container`, `Wrapper` 같은 모호한 이름 지적
- **Props/변수 이름**: boolean은 `is-`/`has-`/`should-` 접두사. 이벤트 핸들러는 `on-`/`handle-`
- **파일/폴더 구조**: 팀 컨벤션 준수. 관련 파일(컴포넌트·스타일·테스트)이 가까이 위치

## 출력 (code-reviewer Part A로 통합)

본 스킬은 단독 출력 형식을 갖지 않는다. `code-reviewer` 에이전트가 본 체크리스트를 적용해 Part A 항목으로 통합한다:

```yaml
part_a:
  items:
    - severity: 🔴 | 🟡
      area: "컴포넌트 설계 / 상태 관리 / 스타일링 / 접근성 / 성능 / 네이밍"
      message: "구체적 설명"
      file: path/to/file
      line: N (선택)
```

### Severity 가이드

- **🔴**: 다른 컴포넌트로 퍼지면 수정 비용 급증. 공통 컴포넌트의 잘못된 인터페이스, 접근성 미준수, 필수 상태 누락
- **🟡**: 코드 품질 저해하나 즉시 퍼지지는 않음. 매직 넘버, 과도한 리렌더링, 모호한 네이밍
- **pass**: 발견 사항 없음
