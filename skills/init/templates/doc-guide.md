# Documentation Management Guide

이 가이드는 `docs/` 디렉토리 내 설계 문서를 버전 관리하기 위한 규칙을 정의한다.
문서를 생성하거나 수정하는 모든 에이전트와 사람은 이 가이드를 따른다.

---

## 디렉토리 구조

```
docs/
├── INDEX.md                        # 문서 목록 및 현재 버전 요약
├── prd/
│   ├── prd.md                      # 최신본 (항상 현재 버전)
│   └── CHANGELOG.md                # 버전별 변경 요약
├── api-design/
│   ├── api-design.md
│   └── CHANGELOG.md
├── db-design/
│   ├── db-design.md
│   └── CHANGELOG.md
└── ...
```

### 규칙

- 문서 카테고리 하나당 하나의 디렉토리를 만든다.
- 디렉토리 이름은 kebab-case를 사용한다.
- 디렉토리 안에는 반드시 `<카테고리명>.md`, `CHANGELOG.md`를 포함한다.
- 카테고리명은 canonical form(표준 형태, 예: `db-design`, `ui-design`)을 따른다. 프로젝트 특성상 canonical form이 직접 적용되지 않는 경우 문서 내용은 재해석된 의미로 작성할 수 있으나, 파일명과 디렉토리명은 바꾸지 않는다. 재해석 시 문서 최상단에 `## 0. 범위 선언` 섹션을 두어 독자가 선입견 없이 읽도록 안내한다.

## Frontmatter 규격

모든 문서의 최상단에 YAML frontmatter를 작성한다.

```yaml
---
title: Payment API Design
version: 2.4.0
created: 2026-03-01
updated: 2026-04-15
---
```

### 필수 필드

| 필드       | 설명                                                |
| ---------- | --------------------------------------------------- |
| `title`    | 문서 제목                                           |
| `version`  | 현재 문서 버전 (semver)                             |
| `updated`  | 마지막 수정일 (YYYY-MM-DD)                          |
| `based_on` | 선행 의존 문서가 있는 경우 필수. 없으면 생략       |

### 선택 필드

| 필드      | 설명        |
| --------- | ----------- |
| `created` | 최초 작성일 |

### `based_on` 상세

`based_on`은 문서 간 의존 관계를 추적하기 위한 필드다. 선행 의존 문서가 **하나 이상 존재하는 문서는 반드시** 이 필드를 기재한다. 의존이 전혀 없는 문서(예: PRD)는 필드를 생략한다.

형식은 `<카테고리>@<버전>` 문자열의 배열을 사용한다:

```yaml
based_on:
  - prd@0.1.0
  - architecture@0.2.0
```

의존 문서가 갱신되면 `based_on`의 버전을 함께 올린다.

### 카테고리별 필수 의존

nidost 기획·설계 체인의 카테고리별 필수 의존은 아래와 같다. 각 spec 스킬은 이 테이블의 필수 의존을 `based_on`에 반드시 포함해야 한다.

| 카테고리         | 필수 의존                                      | 선택 의존       |
| ---------------- | ---------------------------------------------- | --------------- |
| `prd`            | —                                              | —               |
| `user-journey`   | `prd`                                          | —               |
| `architecture`   | `prd`                                          | —               |
| `design-tokens`  | `prd`                                          | —               |
| `ui-design`      | `prd`, `user-journey`, `architecture`          | `design-tokens` |
| `db-design`      | `prd`, `architecture`                          | —               |
| `api-design`     | `prd`, `architecture`, `db-design`             | —               |

선택 의존은 로드된 경우에만 `based_on`에 포함한다.

## 버전 번호 규칙

Semantic Versioning을 따른다: `MAJOR.MINOR.PATCH`

| 변경 수준                | 버전       | 예시          |
| ------------------------ | ---------- | ------------- |
| 구조 변경, 대규모 재작성 | MAJOR 증가 | 1.x.x → 2.0.0 |
| 섹션 추가, 항목 변경     | MINOR 증가 | 2.3.x → 2.4.0 |
| 오타 수정, 문구 다듬기   | PATCH 증가 | 2.4.0 → 2.4.1 |

## 문서 생성 절차

### 1. 디렉토리 및 파일 생성

```bash
mkdir -p docs/api-design
```

`docs/api-design/api-design.md` 파일을 생성하고, frontmatter를 작성한다:

```yaml
---
title: API Design
version: 0.1.0
created: 2026-04-15
updated: 2026-04-15
---
```

### 2. CHANGELOG.md 생성

```markdown
# API Design Changelog

## 0.1.0 (2026-04-15)

- 초안 작성
```

### 3. INDEX.md 업데이트

`docs/INDEX.md`에 새 문서 항목을 추가한다.

### 4. Git commit 및 tag

```bash
git add docs/api-design/
git commit -m "docs(api-design): v0.1.0 - 초안 작성"
git tag doc/api-design/v0.1.0 -m "초안 작성"
```

---

## 문서 수정 절차

### 1. 문서 내용 수정

본문을 수정한다.

### 2. Frontmatter 업데이트

- `version`을 변경 수준에 맞게 올린다.
- `updated`를 오늘 날짜로 변경한다.

### 3. CHANGELOG.md 업데이트 (MAJOR/MINOR 변경만)

PATCH 변경(오타, 포맷팅, 문구 다듬기)은 CHANGELOG에 기록하지 않는다. MAJOR 또는 MINOR 변경인 경우에만 CHANGELOG.md 최상단(`# 제목` 바로 아래)에 새 항목을 추가한다:

```markdown
## 2.4.0 (2026-04-15)

- 웹소켓 실시간 알림 엔드포인트 추가
- 인증 토큰 갱신 플로우 변경
```

### 4. INDEX.md 업데이트

`docs/INDEX.md`에서 해당 문서의 버전 번호와 상태를 갱신한다.

### 5. Git commit 및 tag

```bash
git add docs/api-design/
git commit -m "docs(api-design): v2.4.0 - 웹소켓 알림 엔드포인트 추가"
git tag doc/api-design/v2.4.0 -m "웹소켓 알림 엔드포인트 추가"
```

## Git 컨벤션

### Commit message 형식

```
docs(<카테고리>): v<버전> - <변경 요약>
```

예시:

```
docs(prd): v1.2.11 - 결제 수단 추가 요구사항 반영
docs(api-design): v2.4.0 - 웹소켓 알림 엔드포인트 추가
docs(db-design): v1.0.0 - 초기 스키마 확정
```

### Tag 형식

```
doc/<카테고리>/v<버전>
```

예시:

```
doc/prd/v1.2.11
doc/api-design/v2.4.0
doc/db-design/v1.0.0
```

### 주의사항

- 코드의 릴리스 태그(`v1.0.0`)와 문서 태그(`doc/*/v*`)는 prefix로 구분된다.
- 태그는 반드시 commit과 동시에 생성한다. 나중에 따로 찍지 않는다.
- PATCH 변경(오타, 포맷팅)도 태그를 찍는다. 태그 없는 문서 변경은 허용하지 않는다.

## CHANGELOG.md 작성 규칙

### 형식

```markdown
# <문서 제목> Changelog

## <버전> (<날짜>)

- 변경 내용 1
- 변경 내용 2

## <이전 버전> (<날짜>)

- 변경 내용
```

### 규칙

- MAJOR, MINOR 변경만 기록한다. PATCH 변경(오타, 포맷팅, 문구 다듬기)은 기록하지 않는다.
- 최신 버전이 파일 상단에 위치한다 (역순).
- 각 항목은 한 줄로 요약한다. 상세 내용은 본문 문서를 참조한다.
- git commit message와 동일한 내용을 쓰되, CHANGELOG에는 여러 항목을 나열할 수 있다. commit message는 대표 항목 하나만 쓴다.
- 본문 문서 내에 `## Changelog` 섹션을 만들지 않는다. 변경 이력은 반드시 이 `CHANGELOG.md` 파일로만 관리한다.

## INDEX.md 작성 규칙

`docs/INDEX.md`는 에이전트가 문서를 탐색하는 진입점이다.

### 형식

```markdown
# Documentation Index

| 문서       | 경로                                                 | 버전   | 설명                     |
| ---------- | ---------------------------------------------------- | ------ | ------------------------ |
| PRD        | [prd/prd.md](prd/prd.md)                             | 1.2.11 | 제품 요구사항 정의서     |
| API Design | [api-design/api-design.md](api-design/api-design.md) | 2.4.0  | REST API 명세            |
| DB Design  | [db-design/db-design.md](db-design/db-design.md)     | 1.0.0  | 데이터베이스 스키마 설계 |
```

### 규칙

- 문서가 추가, 삭제, 버전 변경될 때마다 INDEX.md를 함께 갱신한다.
- 경로는 `docs/` 기준 상대 경로를 사용한다.

## 결정 로그 규약

설계 문서(spec-*)는 작성 과정에서 내린 주요 선택과 그 근거를 **결정 로그** 섹션에 기록한다. 결정 로그는 문서의 **마지막 표준 섹션**에 위치하며, 섹션 번호는 문서별로 다르다.

### 표 형식

| 결정 축 | 결정 | 근거 | 제쳐진 후보 |
| ------- | ---- | ---- | ----------- |
| ...     | ...  | ...  | ...         |

### 필드 설명

- **결정 축** — 결정의 대상. 한 단어~한 구 수준으로 짧게 (예: "ID 전략", "버저닝 전략", "내비게이션 패턴")
- **결정** — 최종 선택. 구체적인 값·방식을 그대로 기록한다
- **근거** — 1~2문장. "왜 다른 후보를 제쳤는가"에 초점
- **제쳐진 후보** — 검토했지만 선택하지 않은 대안들. 콤마로 구분

### 자동 선택 표기

PRD·선행 설계 문서에서 **자명하게 도출된 결정**(사용자 인터뷰 없이 자동으로 결정된 축)은 `근거` 컬럼 끝에 `(자동 선택)`을 붙인다. 사용자가 "알아서 해", "추천대로 해주세요" 등으로 인터뷰를 위임한 경우에도 동일하게 표기한다.

예:

```
| ID 전략 | BIGINT AUTO_INCREMENT | 단일 인스턴스·외부 노출 없음 (자동 선택) | UUID, ULID |
```

### 재해석 모드 표기

프로젝트가 재해석 모드(`## 0. 범위 선언` 섹션 추가)로 진입한 경우, 결정 로그 **최상단**에 재해석 결정 축을 추가하고 canonical form을 "제쳐진 후보"로 기록한다. 결정 축 이름은 스킬 성격에 따라 달라진다:

| 스킬              | 재해석 결정 축 이름 | canonical form 예시      |
| ----------------- | ------------------- | ------------------------ |
| spec-architecture | 프로젝트 유형       | 클라우드 런타임 서비스   |
| spec-db-design    | 저장소 유형         | PostgreSQL 관계형 DB     |
| spec-api-design   | 인터페이스 형태     | HTTP REST API            |
| spec-ui-design    | 인터페이스 형태     | 그래픽 사용자 인터페이스 |
| spec-design-tokens| 표현 매체           | 시각적 UI 디자인 토큰    |

### 결정 로그를 가지지 않는 문서

`spec-user-journey`와 `spec-design-tokens`는 별도 결정 로그 섹션을 두지 않는다. 재해석 모드에서의 결정 근거는 `## 0. 범위 선언` 섹션 본문에 서술형으로 기록한다.

## 인터뷰 상한 규약

spec-* 스킬의 STEP 2 대화형 인터뷰는 아래 공통 규약을 따른다. 각 스킬은 이 규약을 참조하고, 결정 축 목록·추천 선정 우선순위·첫 질문 규칙·위임 불가 축 등 스킬별 특이사항만 본문에 기술한다.

### 상한 티어

| 티어     | 권장 상한 | 강제 상한 | 해당 스킬                                                                       |
| -------- | --------- | --------- | ------------------------------------------------------------------------------- |
| Heavy    | 5         | 7         | `spec-architecture`                                                             |
| Standard | 4         | 6         | `spec-design-tokens`, `spec-ui-design`, `spec-db-design`, `spec-api-design`     |

Heavy 티어는 결정 축이 6개 이상이고 후속 모든 스킬의 기반이 되는 스킬을 의미한다. 새 spec 스킬을 추가할 때는 축 수와 영향 범위를 기준으로 티어를 분류한다.

### 정지 조건

아래 조건 중 **하나라도** 만족하면 즉시 인터뷰를 종료하고 STEP 3로 진행한다. 상한 숫자는 최종 안전망이며, 평상시에는 아래 조건으로 자연 종료되어야 한다.

1. 핵심 결정 축이 모두 자동 도출되거나 답변 완료됨
2. 남은 축이 추천 기본값으로 결정 가능한 수준으로 좁혀짐
3. 같은 결정이 다른 표현으로 반복 등장하기 시작함 (진전 없음)
4. 사용자가 위임 신호("알아서 해", "추천대로 해주세요" 등)를 보냄
5. 권장 상한 도달: 남은 축은 자동 결정 가능한지 재평가한 뒤, 가능하면 자동 결정
6. 강제 상한 도달: 남은 축을 모두 추천 기본값으로 자동 결정

### 앵커 패턴

모든 인터뷰 응답은 아래 구조를 따른다:

1. **컨텍스트 요약** — 이전 답변이 현재 단계에 어떻게 연결되는지 한 줄 요약
2. **핵심 질문 1개** — 사용자 전문성 수준에 맞춘 깊이
3. **선택지 2~5개**
   - `(추천)` 태그 + 이유 1~2문장
   - 치명적 리스크는 한 줄 이내 언급
   - 항상 '직접 입력' 포함
4. **대기** — 응답 생성 중단

한 번에 하나의 결정 축만 질문한다. 답변이 충분히 구체적이면 다음 축을 건너뛴다.

### 사용자 위임 처리

사용자가 "알아서 해", "추천대로 해주세요" 등으로 인터뷰를 위임하는 경우:

- 모든 축을 추천 기본값으로 자동 결정한다
- 결정 로그의 `근거` 컬럼에 `(자동 선택)` 표기를 추가한다
- **위임 불가 축**이 지정된 스킬(예: `spec-architecture`의 배포 타깃·백엔드 스택, `spec-design-tokens`의 색상 팔레트)은 해당 축만 질문을 유지하고 답변을 받는다

### 직접 입력 처리

사용자가 선택지 외 의견을 입력하는 경우:

- **비판적 검토** — 실현 가능성과 선행 결정과의 일관성을 평가한다. 맹목적 수용 금지
- **모호함 해결** — 입력이 모호하면 다음 단계로 넘어가기 전 후속 질문 1개로 명확화
- **피벗 브리핑** — 방향이 변하면 영향 범위를 요약한 뒤 진행

### 전문성 중립 질문

인터뷰 질문은 사용자의 전문 분야와 무관하게 답할 수 있도록 설계한다. 개발자가 디자인 결정을 감성·선호로 답하듯, 비개발자도 기술 결정을 비즈니스 맥락·선호로 답할 수 있어야 한다.

**질문 설계 규칙:**

1. 기술 용어를 질문의 전제로 사용하지 않는다. 선택지에 기술 용어를 포함할 수는 있으나 비즈니스 설명을 반드시 병기한다
2. 각 선택지의 `(추천)` 이유는 기술적 이점이 아닌 비즈니스 이점·결과로 표현한다
3. "직접 입력" 선택지에는 "기술명이나 선호를 자유 기술"이라고 안내해, 전문 지식이 있는 사용자가 직접 답변할 수 있는 경로를 열어 둔다
4. 사용자가 기술명·전문 용어로 직접 답변하면 숙련도 신호로 간주하고 이후 질문 수준을 상향한다
5. 사용자가 비즈니스 맥락·감성·선호로 답변하면 AI가 최적의 전문 결정으로 변환하고 결정 로그에 비즈니스 근거와 전문 선택을 함께 기록한다
6. 기술 용어가 선택지에 등장할 때는 한 줄로 풀어 설명을 병기한다 (예: "soft delete = 데이터를 실제로 지우지 않고 '삭제됨' 표시만 남기는 방식")

**유효한 답변의 세 층위:**

| 층위 | 예시 | AI 처리 |
| --- | --- | --- |
| 비즈니스 맥락 | "서버 관리는 안 하고 싶어요" | AI가 관리형 솔루션으로 결정 |
| 선호·감성 | "빠르고 심플한 게 좋아요" | AI가 경량 스택으로 결정 |
| 전문 명시 | "PostgreSQL + Redis" | AI가 그대로 수용, 숙련도 상향 |

세 층위 모두 같은 질문에 대한 유효한 답변이다. AI는 답변의 추상도를 감지해 이후 질문 수준을 조정한다.

## 특정 버전 조회 방법

파일을 직접 읽는다:

cat docs/api-design/api-design.md

```bash
git show doc/api-design/v2.3.1:docs/api-design/api-design.md
```

```bash
git diff doc/api-design/v2.3.1 doc/api-design/v2.4.0 -- docs/api-design/api-design.md
```

```bash
cat docs/api-design/CHANGELOG.md
```

## 체크리스트

문서를 생성하거나 수정할 때, 아래 항목을 모두 확인한다:

- [ ] frontmatter의 `version`이 변경 수준에 맞게 올라갔는가?
- [ ] frontmatter의 `updated`가 오늘 날짜인가?
- [ ] MAJOR/MINOR 변경인 경우 CHANGELOG.md에 변경 항목이 추가되었는가?
- [ ] INDEX.md의 버전이 갱신되었는가?
- [ ] commit message가 `docs(<카테고리>): v<버전> - <요약>` 형식인가?
- [ ] `doc/<카테고리>/v<버전>` 태그가 생성되었는가?
