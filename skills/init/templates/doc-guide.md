# Documentation Management Guide

이 가이드는 `docs/` 디렉토리 내 설계 문서를 버전 관리하기 위한 규칙을 정의한다.
문서를 생성하거나 수정하는 모든 에이전트와 사람은 이 가이드를 따른다.

---

## 디렉토리 구조

```
docs/
├── INDEX.md                        # 문서 목록 및 현재 버전 요약
├── prd/
│   ├── prd.md                      # 최신본 (항상 현재 버전)
│   └── CHANGELOG.md                # 버전별 변경 요약
├── api-design/
│   ├── api-design.md
│   └── CHANGELOG.md
├── db-design/
│   ├── db-design.md
│   └── CHANGELOG.md
└── ...
```

### 규칙

- 문서 카테고리 하나당 하나의 디렉토리를 만든다.
- 디렉토리 이름은 kebab-case를 사용한다.
- 디렉토리 안에는 반드시 `<카테고리명>.md`, `CHANGELOG.md`를 포함한다.
- 카테고리명은 canonical form(표준 형태, 예: `db-design`, `ui-design`)을 따른다. 프로젝트 특성상 canonical form이 직접 적용되지 않는 경우 문서 내용은 재해석된 의미로 작성할 수 있으나, 파일명과 디렉토리명은 바꾸지 않는다. 재해석 시 문서 최상단에 `## 0. 범위 선언` 섹션을 두어 독자가 선입견 없이 읽도록 안내한다.

## Frontmatter 규격

모든 문서의 최상단에 YAML frontmatter를 작성한다.

```yaml
---
title: Payment API Design
version: 2.4.0
created: 2026-03-01
updated: 2026-04-15
---
```

### 필수 필드

| 필드       | 설명                                                |
| ---------- | --------------------------------------------------- |
| `title`    | 문서 제목                                           |
| `version`  | 현재 문서 버전 (semver)                             |
| `updated`  | 마지막 수정일 (YYYY-MM-DD)                          |
| `based_on` | 선행 의존 문서가 있는 경우 필수. 없으면 생략       |

### 선택 필드

| 필드      | 설명        |
| --------- | ----------- |
| `created` | 최초 작성일 |

### `based_on` 상세

`based_on`은 문서 간 의존 관계를 추적하기 위한 필드다. 선행 의존 문서가 **하나 이상 존재하는 문서는 반드시** 이 필드를 기재한다. 의존이 전혀 없는 문서(예: PRD)는 필드를 생략한다.

형식은 `<카테고리>@<버전>` 문자열의 배열을 사용한다:

```yaml
based_on:
  - prd@0.1.0
  - architecture@0.2.0
```

의존 문서가 갱신되면 `based_on`의 버전을 함께 올린다.

### 카테고리별 필수 의존

nidost 기획·설계 체인의 카테고리별 필수 의존은 아래와 같다. 각 spec 스킬은 이 테이블의 필수 의존을 `based_on`에 반드시 포함해야 한다.

| 카테고리         | 필수 의존                                                       |
| ---------------- | --------------------------------------------------------------- |
| `prd`            | —                                                               |
| `user-journey`   | `prd`                                                           |
| `architecture`   | `prd`                                                           |
| `design-tokens`  | `prd`                                                           |
| `ui-design`      | `prd`, `user-journey`, `architecture`, `design-tokens`          |
| `db-design`      | `prd`, `architecture`                                           |
| `api-design`     | `prd`, `architecture`, `db-design`                              |

## 버전 번호 규칙

Semantic Versioning을 따른다: `MAJOR.MINOR.PATCH`

| 변경 수준                | 버전       | 예시          |
| ------------------------ | ---------- | ------------- |
| 구조 변경, 대규모 재작성 | MAJOR 증가 | 1.x.x → 2.0.0 |
| 섹션 추가, 항목 변경     | MINOR 증가 | 2.3.x → 2.4.0 |
| 오타 수정, 문구 다듬기   | PATCH 증가 | 2.4.0 → 2.4.1 |

### 카테고리별 변경 수준 예시

| 카테고리       | PATCH 예시              | MINOR 예시                       | MAJOR 예시                         |
| -------------- | ----------------------- | -------------------------------- | ---------------------------------- |
| prd            | 오타 수정               | 페르소나 추가, 기능 추가         | 핵심 목표 변경, 타깃 피벗          |
| user-journey   | 문구 다듬기             | 시나리오 추가                    | 주 페르소나 전환                   |
| architecture   | 컴포넌트 설명 보강      | 부가 컴포넌트 추가               | 백엔드·배포 타깃 전환              |
| design-tokens  | HEX 값 미세 조정        | 토큰 추가                        | 색상 팔레트 전면 교체              |
| ui-design      | 레이아웃 문구 다듬기    | 화면 추가                        | 내비게이션 패턴 전환               |
| db-design      | 컬럼 설명 보강          | 테이블·인덱스 추가               | 저장소 유형 전환 (RDBMS↔NoSQL)     |
| api-design     | 엔드포인트 설명 다듬기  | 엔드포인트 추가                  | 인증 방식·버저닝 전략 변경         |

## 문서 생성 절차

### 1. 디렉토리 및 파일 생성

```bash
mkdir -p docs/api-design
```

`docs/api-design/api-design.md` 파일을 생성하고, frontmatter를 작성한다:

```yaml
---
title: API Design
version: 0.1.0
created: 2026-04-15
updated: 2026-04-15
---
```

### 2. CHANGELOG.md 생성

```markdown
# API Design Changelog

## 0.1.0 (2026-04-15)

- 초안 작성
```

### 3. INDEX.md 업데이트

`docs/INDEX.md`에 새 문서 항목을 추가한다.

### 4. Lock (commit + tag)

표준 경로: `/nidost:spec-lock api-design`. 수동 수행 시 형식은 「Git 컨벤션」을 참조한다.

---

## 문서 수정 절차

### 1. 문서 내용 수정

본문을 수정한다.

### 2. Frontmatter 업데이트

- `version`을 변경 수준에 맞게 올린다.
- `updated`를 오늘 날짜로 변경한다.

### 3. CHANGELOG.md 업데이트 (MAJOR/MINOR 변경만)

PATCH 변경(오타, 포맷팅, 문구 다듬기)은 CHANGELOG에 기록하지 않는다. MAJOR 또는 MINOR 변경인 경우에만 CHANGELOG.md 최상단(`# 제목` 바로 아래)에 새 항목을 추가한다:

```markdown
## 2.4.0 (2026-04-15)

- 웹소켓 실시간 알림 엔드포인트 추가
- 인증 토큰 갱신 플로우 변경
```

### 4. INDEX.md 업데이트

`docs/INDEX.md`에서 해당 문서의 버전 번호와 상태를 갱신한다.

### 5. Lock (commit + tag)

표준 경로: `/nidost:spec-lock api-design`. 수동 수행 시 형식은 「Git 컨벤션」을 참조한다.

## Lock 상태 수정 프로토콜 (스킬 자동화용)

spec-* 스킬이 **Lock 상태**(해당 버전 태그가 이미 존재하는 상태)의 기존 문서를 수정할 때 따르는 공통 프로토콜이다. 위의 「문서 수정 절차」가 사람을 위한 수동 체크리스트라면, 이 프로토콜은 스킬이 참조하는 자동화 규약이다.

Working 상태 편집은 별도 「Working 상태 편집 규칙」을 따른다 (파일 직접 편집 가능, 버전·CHANGELOG 변경 없음).

### STEP 0 선택지

Lock 상태의 기존 문서가 발견되면 아래 두 가지를 제시한다:

1. **수정** — 기존 문서를 읽고 특정 섹션만 재생성하거나 전체를 재생성한다. 새 버전의 Working 상태로 진입한다
2. **종료** — 아무것도 하지 않는다

초기화(전체 삭제)가 정말 필요하면 사용자가 수동으로 `rm -rf docs/<category>/` 후 스킬을 재호출한다.

### 수정 모드 동작

수정(1번)을 선택하면 아래 순서로 진행한다:

#### 1. 기존 문서 로드

기존 문서의 frontmatter(version, based_on)와 본문 전체를 읽는다.

#### 2. 상위 문서 변경 감지

`based_on`에 기록된 각 의존 문서의 현재 frontmatter version과 비교한다.

- 버전이 올라간 의존 문서가 있으면 → **변경 감지 보고** 출력
- 없으면 → "상위 문서 변경 없음" 보고 후 3단계로 진행

변경 감지 보고 형식:

```
상위 문서 변경 감지:
  - prd: 0.1.0 → 0.2.0 (변경됨)
  - architecture: 0.1.0 (동일)
```

#### 3. 수정 대상 섹션 결정

변경된 상위 문서가 있으면 영향받을 수 있는 섹션을 자동 제안한다. 사용자에게 확인한다:

```
PRD v0.2.0 변경으로 영향받을 수 있는 섹션:
  - §2 테이블 정의
  - §4 관계 정의

위 섹션을 갱신하시겠어요? 다른 섹션도 추가하시겠어요?

1. 제안된 섹션만 갱신
2. 섹션 추가 선택 (자유 입력)
3. 전체 재생성
```

변경된 상위 문서가 없으면 섹션 목록을 보여주고 사용자가 직접 선택한다:

```
현재 문서의 섹션:
  §1. ERD 다이어그램
  §2. 테이블 정의
  §3. 인덱스 전략
  ...

어떤 섹션을 수정하시겠어요? (번호, 예: 2,3)
전체를 재생성하려면 "전체"를 입력하세요.
```

#### 4. 섹션 재생성

선택된 섹션을 해당 스킬의 기존 STEP 3 작성 로직으로 재생성한다. 선택되지 않은 섹션은 기존 내용을 그대로 유지한다. 재생성 시 현재 상위 문서(최신 버전)를 입력으로 사용한다.

"전체 재생성" 선택 시: 기존 문서의 결정 로그와 전체 내용이 컨텍스트에 있으므로, 이전 결정이 자연스럽게 기본값으로 반영된다.

#### 5. Frontmatter 갱신

- `version`: 변경 수준에 따라 올린다
  - 섹션 추가·내용 변경 → MINOR 증가 (예: 0.1.0 → 0.2.0)
  - 구조 변경·재해석 전환·전체 재생성 → MAJOR 증가 (예: 0.1.0 → 1.0.0)
  - 오타·문구 수정 → PATCH 증가 (예: 0.1.0 → 0.1.1)
- `updated`: 오늘 날짜
- `based_on`: 현재 상위 문서 버전으로 갱신

#### 6. CHANGELOG·INDEX 갱신

- CHANGELOG.md: MAJOR/MINOR 변경 시 새 항목 추가 (PATCH는 생략)
- INDEX.md: 버전 번호 갱신

#### 7. 완료 보고

완료 요약을 출력한 뒤 「작성 완료 Lifecycle 프롬프트」 절차를 그대로 수행한다 (신규 생성 시와 동일, 버전 번호만 다름).

## 문서 수명 주기

모든 spec-* 문서는 **Working ↔ Lock** 두 상태를 오가며 진화한다.

### 두 상태

- **Working** — 편집 가능 상태. 같은 버전 내에서 자유롭게 이터레이션. 아직 커밋·태그 없음
- **Lock** — 커밋 + 태그 (`doc/<카테고리>/v<버전>`) 완료. 불변 상태

### Lock 판별 기준

해당 카테고리와 버전에 태그가 존재하는지로 판별한다:

```bash
git tag --list "doc/<카테고리>/v<version>" | grep -q . && echo "locked" || echo "working"
```

### 상태 전이

```
신규 생성 → Working (v0.1.0)
Working + commit + tag → Lock
Lock + 수정 시작 → Working (v0.2.0)
Working + commit + tag → Lock
... (반복)
```

**Revision**은 별도 상태가 아니라 **새 버전의 Working 단계**이다. Lock된 문서를 수정하기 시작하면 새 버전의 Working으로 진입한다.

### spec 스킬 재실행 시 동작

| 파일 | 태그 | 상태 | 동작 |
| ---- | ---- | ---- | ---- |
| X    | —    | —    | 신규 생성 (v0.1.0 Working) |
| O    | X    | Working | 이어서 편집 (현재 버전 유지) |
| O    | O    | Lock    | 「Lock 상태 수정 프로토콜」 진입 |

## Working 상태 편집 규칙

Working 상태의 문서는 자유롭게 편집 가능하다. 이터레이션 중에는 다음 규칙을 따른다:

- **파일 직접 편집 허용**: 스킬 재호출 없이 텍스트 에디터로 수정 가능
- **스킬 재호출 가능**: "이어서 편집" 모드로 진입. 특정 섹션을 다시 생성하거나 전체를 다시 쓸 수 있음. 이 경우에도 버전은 유지됨
- **버전 불변**: Working 기간 전체에 걸쳐 frontmatter의 `version`은 그대로 (예: v0.1.0 전체 작업 기간 동안 0.1.0)
- **CHANGELOG 불변**: 해당 버전의 CHANGELOG 항목은 최초 생성 시 작성되고, Working 중에는 필요 시 내용만 다듬되 새 버전 항목은 추가하지 않음
- **INDEX 불변**: Working 중 버전 번호 변경 없으므로 INDEX도 그대로
- **git 미커밋 권장**: Lock 시점까지 커밋하지 않는 것을 권장. 필요 시 working 커밋은 가능하지만 태그는 붙이지 않음

### Lock 수행

Working → Lock 전환은 두 가지 방법으로 수행한다:

1. **유틸리티 스킬 사용 (권장)**: `/nidost:spec-lock <카테고리>` — commit + annotated tag를 자동 수행
2. **수동 커밋**: 「Git 컨벤션」의 형식대로 `git add` → `git commit` → `git tag -a`

## 묵시적 Lock 유도

spec-* 스킬은 실행 시작 시점에 **필수 선행 문서들의 상태**를 검사한다. 필수 선행 문서 중 하나라도 Working 상태이면 아래와 같이 사용자에게 Lock을 권유한다:

```
⚠️ `<선행 카테고리>`가 v<version> Working 상태입니다.
   Lock하지 않으면 기반 버전이 불명확한 채로 이 문서가 작성됩니다.

1. 지금 Lock (/nidost:spec-lock <선행 카테고리> 실행)
2. 현재 Working 상태 그대로 진행 (권장하지 않음)
3. 종료
```

- **1번 선택**: spec-lock 스킬을 호출해 선행 문서를 Lock → 완료 후 현재 스킬 STEP 0 계속
- **2번 선택**: 현재 스킬 계속 진행. `based_on`에 해당 버전을 기록하되, 주석으로 "Working 참조" 플래그 추가 가능
- **3번 선택**: 종료

## 작성 완료 Lifecycle 프롬프트

spec-* 스킬이 본문 작성과 INDEX·CHANGELOG 갱신을 마친 뒤, STEP 4에서 완료 요약 직후 이 프롬프트로 사용자에게 다음 동작을 묻는다. 모든 spec-* 스킬이 이 절차를 공유한다.

### 3-way 프롬프트

완료 요약 블록 바로 아래에 다음을 출력한다:

```
이 문서를 어떻게 할까요?

1. 작성 완료 (문서 Lock) — `/nidost:spec-lock <카테고리>` 호출
2. Working 유지하고 종료 — 이후 자유롭게 편집하거나 `/nidost:spec-lock`으로 Lock 가능
3. 이 문서 추가 편집 — 섹션 단위로 이어서 다듬기

팁: 여러 문서를 연이어 작성 중이면 모두 Working으로 두고
    마지막에 `/nidost:spec-lock`으로 한 번에 Lock하는 편이 효율적입니다.
```

### 1번 선택 — Lock

`spec-lock` 스킬을 단일 모드로 호출한다: `/nidost:spec-lock <카테고리>`. Lock 수행(commit + annotated tag)은 spec-lock이 전담하며, 본 스킬은 호출 후 종료한다.

### 2번 선택 — Working 유지 종료

스킬을 종료한다. 문서는 Working 상태로 남고 CHANGELOG는 초기 생성본 그대로 유지한다. 이후 사용자가 같은 스킬을 재호출하면 「Working 상태 편집 규칙」의 "이어서 편집" 모드로 진입한다.

### 3번 선택 — 추가 편집

섹션 단위 편집 루프에 진입한다. 루프는 아래 순서로 1회 반복한다.

#### 3-1. 섹션 목록 제시

문서의 `## ` 헤딩을 추출해 번호 리스트로 표시한다:

```bash
grep -n '^## ' docs/<카테고리>/<카테고리>.md
```

```
현재 문서의 섹션:
  1. §0. 범위 선언
  2. §1. ...
  ...

어느 부분을 수정하고 싶으신가요? (번호 입력, 여러 개는 콤마 구분)
```

#### 3-2. 자유 입력 수집

선택된 섹션의 현재 내용을 보여주고 사용자의 수정 요청을 자유 입력으로 받는다 (예: "톤을 더 단호하게", "페르소나 2의 고통점을 한 줄 추가").

#### 3-3. Diff 제안 및 승인

사용자 요청을 바탕으로 수정안을 생성해 before/after diff로 제시한다:

```
1. 적용
2. 다시 요청 (수정 요청 재입력)
3. 취소 (변경 없이 섹션 선택으로 돌아가기)
```

"1. 적용"을 선택하면 파일을 실제로 편집하고, 세션 내부 메모에 한 줄 요약을 누적한다 (CHANGELOG는 아직 건드리지 않음).

#### 3-4. 3-way 프롬프트 복귀

편집 1회가 끝나면 다시 3-way 프롬프트를 표시한다. 사용자가 1번 또는 2번을 선택할 때까지 반복한다.

### CHANGELOG 처리

3번 경로의 편집은 CHANGELOG를 실시간 갱신하지 않는다. 루프 종료 시점(1번 또는 2번 선택 순간)에 누적 메모를 요약해 **한 번만** 정리한다:

1. 누적된 편집 요약을 사용자에게 보여준다
2. 사용자가 최종 불릿을 확인·다듬는다
3. `## <현재 버전>` 블록에 반영한다

이후 1번이면 spec-lock을 호출하고, 2번이면 그대로 종료한다. 편집 내용이 사소해 CHANGELOG에 반영할 필요가 없다고 판단되면 "변경 없음"으로 건너뛸 수 있다.

### 버전 불변

3번 경로의 편집은 **현재 Working 버전**을 유지한다. 버전 번호를 올리지 않는다. 버전 승격은 오직 「Lock 상태 수정 프로토콜」(새 버전 Working 진입)에서만 발생한다.

## 연쇄 변경 정책

상위 문서의 변경이 하위 문서에 미치는 영향은 bump level에 따라 차등 적용된다.

| Level | 변경 예시                         | 하위 문서 영향   | 동작                                                         |
| ----- | --------------------------------- | ---------------- | ------------------------------------------------------------ |
| PATCH | 오타, 문구 다듬기                 | 없음             | 하위 문서 변경 불필요. `based_on` 갱신은 선택적              |
| MINOR | 섹션·항목 추가, 부분 보강         | 잠재적 영향      | 하위 스킬 다음 실행 시 "갱신?" 안내. 사용자가 스킵하면 유지  |
| MAJOR | 구조 변경, 핵심 결정 번복         | 필수 재검토      | 하위 스킬 실행 시 강한 경고. 재검토 권장                     |

감지 시점: 하위 스킬이 Lock 상태 수정 프로토콜의 STEP 2(상위 문서 변경 감지)에서 `based_on` 버전과 현재 상위 버전을 비교한다.

### 하위 문서 영향 통지

상위 문서가 MINOR 또는 MAJOR로 올라간 경우, 상위 스킬의 완료 보고에 하위 영향을 포함한다:

```
✅ <카테고리> v<new version> 완료

  상위 변경으로 영향받을 수 있는 하위 문서:
    - <하위 카테고리1> v<version> (MINOR 영향 — 다음 실행 시 갱신 안내)
    - <하위 카테고리2> v<version> (MAJOR 영향 — 재검토 권장)
```

---

## Coordinated Update

하위 스킬 작업 중에 상위 문서 수정이 필요함을 깨달은 경우의 워크플로우:

1. 현재 스킬 일시 중단 (Working 상태로 두고 종료)
2. 상위 스킬을 「Lock 상태 수정 프로토콜」 또는 Working 편집으로 호출해 수정
3. 상위 문서를 새 버전 Working → Lock 완료 (`spec-lock`)
4. 현재 스킬 재개 (또는 재실행) → 업데이트된 상위 버전을 `based_on`에 반영
5. 각 문서는 독립적으로 commit + tag 되므로 git 히스토리는 시간순으로 깔끔하게 남음

현재 스킬이 이미 Working 상태로 진입해 있다면 그대로 두고 상위만 수정해도 된다. 재개 시 "이어서 편집" 모드로 진입해 상위 변경 부분만 반영하면 됨.

## Git 컨벤션

### 작업 라인

spec phase의 commit·tag는 **`develop` 브랜치에서 수행**한다. `main`에 직접 docs commit·tag를 만들지 않는다 (git-flow의 "main 직접 커밋 금지" 정책).

각 spec-* 스킬과 `spec-lock`은 STEP 0 첫머리(`0-pre`)에서 현재 브랜치를 검사한다:

- 현재 브랜치가 `main` → `develop`으로 자동 checkout. `main`에 미커밋 변경이 있으면 중단한다 (자동 stash 안 함).
- `develop`, `feature/*`, `release/*`, `hotfix/*`, 그 외 임의 브랜치 → 그대로 진행. 사용자가 의도적으로 다른 라인에서 docs PATCH를 수행하는 시나리오(예: dev 중 PRD 오타 수정을 `feature/<id>`에서, 긴급 spec 보정을 `hotfix/<id>`에서)를 침해하지 않는다.

doc tag는 `main`에서 직접 만들어지지 않으므로 항상 머지 경로를 통해 `main`에 도달한다:

- spec baseline → `develop`에서 commit + tag → 첫 `release/x.y.z` 머지로 `main` 도달
- dev 중 spec PATCH → `feature/<id>` → `develop` → release 머지
- hotfix 중 spec PATCH → `hotfix/<id>` → main+develop 머지

### Commit message 형식

Subject-only (최소 형식):

```
docs(<카테고리>): <영어 imperative subject>
```

Subject + Body (권장 형식, CHANGELOG 현재 버전 블록 본문 포함):

```
docs(<카테고리>): <영어 imperative subject>

<CHANGELOG의 `## <버전>` 블록 본문 전체>
```

예시:

```
docs(prd): reflect new payment requirement

- add new payment method requirement
- update policy matrix to v2
```

```
docs(api-design): add websocket notification endpoints
```

### 언어

커밋 Subject와 Body는 **항상 영어**로 작성한다. CHANGELOG.md 파일 원본은 프로젝트의 도메인 언어(예: 한국어)를 유지할 수 있으며, `/nidost:spec-lock`이 커밋 생성 시점에 번역을 수행한다.

### Subject 규칙

- 영어 imperative (동사 원형으로 시작, 소문자, 마침표 없음)
- 50자 이내 권장
- 버전 문자열은 subject에 포함하지 않는다 (태그에 이미 인코딩됨)
- Breaking change는 `!` 접미사: `docs(<카테고리>)!: <subject>`
- 본문을 작성할 수 없는 fallback 상황에는 `lock v<버전>` 사용

### Tag 형식

annotated tag를 사용하고, 메시지는 commit message와 동일하게 구성한다:

```
doc/<카테고리>/v<버전>
```

예시:

```
doc/prd/v1.2.11
doc/api-design/v2.4.0
doc/db-design/v1.0.0
```

### 권장 실행 경로

이 형식을 맞추기 위한 **표준 실행 경로는 `/nidost:spec-lock`** 스킬이다. 사람 수동으로 수행하는 경우는 아래 명령을 참고한다:

```bash
git add docs/<카테고리>/ docs/INDEX.md
git commit -m "docs(<카테고리>): <subject>" -m "<body>"
git tag -a doc/<카테고리>/v<버전> -m "docs(<카테고리>): <subject>" -m "<body>"
```

### 주의사항

- 코드의 릴리스 태그(`v1.0.0`)와 문서 태그(`doc/*/v*`)는 prefix로 구분된다.
- 태그는 반드시 commit과 동시에 생성한다. 나중에 따로 찍지 않는다.
- PATCH 변경(오타, 포맷팅)도 태그를 찍는다. 태그 없는 문서 변경은 허용하지 않는다.

## CHANGELOG.md 작성 규칙

### 형식

```markdown
# <문서 제목> Changelog

## <버전> (<날짜>)

- 변경 내용 1
- 변경 내용 2

## <이전 버전> (<날짜>)

- 변경 내용
```

### 규칙

- MAJOR, MINOR 변경만 기록한다. PATCH 변경(오타, 포맷팅, 문구 다듬기)은 기록하지 않는다.
- 최신 버전이 파일 상단에 위치한다 (역순).
- 각 항목은 한 줄로 요약한다. 상세 내용은 본문 문서를 참조한다.
- git commit message와 동일한 내용을 쓰되, CHANGELOG에는 여러 항목을 나열할 수 있다. commit message는 대표 항목 하나만 쓴다.
- 본문 문서 내에 `## Changelog` 섹션을 만들지 않는다. 변경 이력은 반드시 이 `CHANGELOG.md` 파일로만 관리한다.

# Documentation Index

| ---------- | ---------------------------------------------------- | ------ | ------------------------ |

### 규칙

- 문서가 추가, 삭제, 버전 변경될 때마다 INDEX.md를 함께 갱신한다.
- 경로는 `docs/` 기준 상대 경로를 사용한다.

## 결정 로그 규약

설계 문서(spec-*)는 작성 과정에서 내린 주요 선택과 그 근거를 **결정 로그** 섹션에 기록한다. 결정 로그는 문서의 **마지막 표준 섹션**에 위치하며, 섹션 번호는 문서별로 다르다.

### 표 형식

| 결정 축 | 결정 | 근거 | 제쳐진 후보 |
| ------- | ---- | ---- | ----------- |
| ...     | ...  | ...  | ...         |

### 필드 설명

- **결정 축** — 결정의 대상. 한 단어~한 구 수준으로 짧게 (예: "ID 전략", "버저닝 전략", "내비게이션 패턴")
- **결정** — 최종 선택. 구체적인 값·방식을 그대로 기록한다
- **근거** — 1~2문장. "왜 다른 후보를 제쳤는가"에 초점
- **제쳐진 후보** — 검토했지만 선택하지 않은 대안들. 콤마로 구분

### 자동 선택 표기

PRD·선행 설계 문서에서 **자명하게 도출된 결정**(사용자 인터뷰 없이 자동으로 결정된 축)은 `근거` 컬럼 끝에 `(자동 선택)`을 붙인다. 사용자가 "알아서 해", "추천대로 해주세요" 등으로 인터뷰를 위임한 경우에도 동일하게 표기한다.

예:

```
| ID 전략 | BIGINT AUTO_INCREMENT | 단일 인스턴스·외부 노출 없음 (자동 선택) | UUID, ULID |
```

### 재해석 모드 표기

프로젝트가 재해석 모드(`## 0. 범위 선언` 섹션 추가)로 진입한 경우, 결정 로그 **최상단**에 재해석 결정 축을 추가하고 canonical form을 "제쳐진 후보"로 기록한다. 결정 축 이름은 스킬 성격에 따라 달라진다:

| 스킬              | 재해석 결정 축 이름 | canonical form 예시      |
| ----------------- | ------------------- | ------------------------ |
| spec-architecture | 프로젝트 유형       | 클라우드 런타임 서비스   |
| spec-db-design    | 저장소 유형         | PostgreSQL 관계형 DB     |
| spec-api-design   | 인터페이스 형태     | HTTP REST API            |
| spec-ui-design    | 인터페이스 형태     | 그래픽 사용자 인터페이스 |
| spec-design-tokens| 표현 매체           | 시각적 UI 디자인 토큰    |

### 결정 로그를 가지지 않는 문서

`spec-user-journey`와 `spec-design-tokens`는 별도 결정 로그 섹션을 두지 않는다. 재해석 모드에서의 결정 근거는 `## 0. 범위 선언` 섹션 본문에 서술형으로 기록한다.

## 재해석 모드의 두 형태

재해석 모드는 프로젝트 특성에 따라 두 가지 형태 중 하나로 작성된다. 어느 형태든 **문서 자체는 항상 존재**하며 INDEX·CHANGELOG·버전 체계는 일관되게 유지된다.

### A1형: 재해석 작성 (Meaningful Adaptation)

**언제**: canonical form이 적용되지 않지만 **다른 형태의 의미 있는 대체 관심사가 존재**할 때.

**예시**: 클라이언트 완결형 모바일 앱의 `spec-db-design` → Catalog(JSON 번들) + Device Store(AsyncStorage) 스키마로 재해석.

**작성 분량**:
- `## 0. 범위 선언`: 2~4단락 (canonical form 미적용 이유, 대체 관심사, 표준 섹션 매핑)
- `## 1. ~ ## N.` (표준 섹션): 모두 재해석된 의미로 채움
- 결정 로그(존재하는 스킬에 한함): 재해석 결정 축 + 나머지 결정 모두 정상 기록

### A2형: 스킵 선언 (Documented Skip)

**언제**: canonical form이 적용되지 않고 **의미 있는 대체 관심사도 없을 때**.

**예시**: 순수 라이브러리 SDK의 `spec-ui-design` → "외부 UI 없음, 작성 불필요"; 헤드리스 백엔드의 `spec-design-tokens` → "시각·출력 표현 없음".

**작성 분량**:
- `## 0. 범위 선언`: 한 단락 ("이 단계는 이 프로젝트에서 작성을 스킵한다. 사유: ..." 형식)
- `## 1. ~ ## N.` (표준 섹션): 각 섹션 제목은 유지하되 내용은 "이 프로젝트에서는 해당 없음. 결정 근거는 §0 참조"로 한 줄 축약
- 결정 로그(존재하는 스킬에 한함): 재해석 결정 축(canonical form vs 스킵) 1행만 기록. 나머지 축은 "N/A"로 표기 또는 생략

### 형태 선택 기준

> **"이 프로젝트에서 이 단계가 다루는 관심사를 다른 형태로라도 의식적으로 결정해야 하는가?"**
>
> - Yes → **A1형 (재해석 작성)**
> - No → **A2형 (스킵 선언)**

A2형도 "왜 작성하지 않는가"를 명시한 **결정 기록**이라는 점에서 가치가 있다. 후속 합류자가 git log에서 "이 단계는 왜 비어 있나?"를 묻지 않게 된다.

### 형태 전환

프로젝트가 진화하면 A2형이 A1형으로, 또는 canonical form으로 승격될 수 있다. 예: v0.1.0에서 A2형으로 작성한 `spec-db-design`이 v0.2.0에서 백엔드 추가로 canonical form으로 전환. 이 전환은 「문서 수정 프로토콜」의 수정 모드에서 수행하고 MAJOR 버전 업으로 처리한다.

## 인터뷰 상한 규약

spec-* 스킬의 STEP 2 대화형 인터뷰는 아래 공통 규약을 따른다. 각 스킬은 이 규약을 참조하고, 결정 축 목록·추천 선정 우선순위·첫 질문 규칙·위임 불가 축 등 스킬별 특이사항만 본문에 기술한다.

### 상한 티어

| 티어     | 권장 상한 | 강제 상한 | 해당 스킬                                                                       |
| -------- | --------- | --------- | ------------------------------------------------------------------------------- |
| Heavy    | 5         | 7         | `spec-architecture`                                                             |
| Standard | 4         | 6         | `spec-design-tokens`, `spec-ui-design`, `spec-db-design`, `spec-api-design`     |

Heavy 티어는 결정 축이 6개 이상이고 후속 모든 스킬의 기반이 되는 스킬을 의미한다. 새 spec 스킬을 추가할 때는 축 수와 영향 범위를 기준으로 티어를 분류한다.

### 정지 조건

아래 조건 중 **하나라도** 만족하면 즉시 인터뷰를 종료하고 STEP 3로 진행한다. 상한 숫자는 최종 안전망이며, 평상시에는 아래 조건으로 자연 종료되어야 한다.

1. 핵심 결정 축이 모두 자동 도출되거나 답변 완료됨
2. 남은 축이 추천 기본값으로 결정 가능한 수준으로 좁혀짐
3. 같은 결정이 다른 표현으로 반복 등장하기 시작함 (진전 없음)
4. 사용자가 위임 신호("알아서 해", "추천대로 해주세요" 등)를 보냄
5. 권장 상한 도달: 남은 축은 자동 결정 가능한지 재평가한 뒤, 가능하면 자동 결정
6. 강제 상한 도달: 남은 축을 모두 추천 기본값으로 자동 결정

### 앵커 패턴

모든 인터뷰 응답은 아래 구조를 따른다:

1. **컨텍스트 요약** — 이전 답변이 현재 단계에 어떻게 연결되는지 한 줄 요약
2. **핵심 질문 1개** — 사용자 전문성 수준에 맞춘 깊이
3. **선택지 2~5개**
   - `(추천)` 태그 + 이유 1~2문장
   - 치명적 리스크는 한 줄 이내 언급
   - 항상 '직접 입력' 포함
4. **대기** — 응답 생성 중단

한 번에 하나의 결정 축만 질문한다. 답변이 충분히 구체적이면 다음 축을 건너뛴다.

### 사용자 위임 처리

사용자가 "알아서 해", "추천대로 해주세요" 등으로 인터뷰를 위임하는 경우:

- 모든 축을 추천 기본값으로 자동 결정한다
- 결정 로그의 `근거` 컬럼에 `(자동 선택)` 표기를 추가한다
- **위임 불가 축**이 지정된 스킬(예: `spec-architecture`의 배포 타깃·백엔드 스택, `spec-design-tokens`의 색상 팔레트)은 해당 축만 질문을 유지하고 답변을 받는다

### 직접 입력 처리

사용자가 선택지 외 의견을 입력하는 경우:

- **비판적 검토** — 실현 가능성과 선행 결정과의 일관성을 평가한다. 맹목적 수용 금지
- **모호함 해결** — 입력이 모호하면 다음 단계로 넘어가기 전 후속 질문 1개로 명확화
- **피벗 브리핑** — 방향이 변하면 영향 범위를 요약한 뒤 진행

### 전문성 중립 질문

인터뷰 질문은 사용자의 전문 분야와 무관하게 답할 수 있도록 설계한다. 개발자가 디자인 결정을 감성·선호로 답하듯, 비개발자도 기술 결정을 비즈니스 맥락·선호로 답할 수 있어야 한다.

**질문 설계 규칙:**

1. 기술 용어를 질문의 전제로 사용하지 않는다. 선택지에 기술 용어를 포함할 수는 있으나 비즈니스 설명을 반드시 병기한다
2. 각 선택지의 `(추천)` 이유는 기술적 이점이 아닌 비즈니스 이점·결과로 표현한다
3. "직접 입력" 선택지에는 "기술명이나 선호를 자유 기술"이라고 안내해, 전문 지식이 있는 사용자가 직접 답변할 수 있는 경로를 열어 둔다
4. 사용자가 기술명·전문 용어로 직접 답변하면 숙련도 신호로 간주하고 이후 질문 수준을 상향한다
5. 사용자가 비즈니스 맥락·감성·선호로 답변하면 AI가 최적의 전문 결정으로 변환하고 결정 로그에 비즈니스 근거와 전문 선택을 함께 기록한다
6. 기술 용어가 선택지에 등장할 때는 한 줄로 풀어 설명을 병기한다 (예: "soft delete = 데이터를 실제로 지우지 않고 '삭제됨' 표시만 남기는 방식")

**유효한 답변의 세 층위:**

| 층위 | 예시 | AI 처리 |
| --- | --- | --- |
| 비즈니스 맥락 | "서버 관리는 안 하고 싶어요" | AI가 관리형 솔루션으로 결정 |
| 선호·감성 | "빠르고 심플한 게 좋아요" | AI가 경량 스택으로 결정 |
| 전문 명시 | "PostgreSQL + Redis" | AI가 그대로 수용, 숙련도 상향 |

세 층위 모두 같은 질문에 대한 유효한 답변이다. AI는 답변의 추상도를 감지해 이후 질문 수준을 조정한다.

## 특정 버전 조회 방법

### 최신 버전 확인

파일을 직접 읽는다:

```bash
cat docs/api-design/api-design.md
```

### 특정 과거 버전 확인

Git tag를 사용한다:

```bash
git show doc/api-design/v2.3.1:docs/api-design/api-design.md
```

### 두 버전 간 차이 확인

```bash
git diff doc/api-design/v2.3.1 doc/api-design/v2.4.0 -- docs/api-design/api-design.md
```

### 변경 이력 요약 확인

```bash
cat docs/api-design/CHANGELOG.md
```

## 체크리스트

문서를 생성하거나 수정할 때, 아래 항목을 모두 확인한다:

- [ ] frontmatter의 `version`이 변경 수준에 맞게 올라갔는가?
- [ ] frontmatter의 `updated`가 오늘 날짜인가?
- [ ] MAJOR/MINOR 변경인 경우 CHANGELOG.md에 변경 항목이 추가되었는가?
- [ ] INDEX.md의 버전이 갱신되었는가?
- [ ] commit message가 `docs(<카테고리>): <영어 imperative subject>` 형식인가? (Body는 CHANGELOG 블록, 상세는 「Git 컨벤션」 참조)
- [ ] `doc/<카테고리>/v<버전>` annotated 태그가 생성되었는가?
